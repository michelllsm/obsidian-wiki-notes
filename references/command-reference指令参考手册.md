## 1. day / 每日记录

### 触发方式
`day` · `day 今天学了xxx` · `day 只记关键事项` · `每日` · `日报`

### 执行流程
1. **读取上下文**
   - 当前对话内容
   - `~/.workbuddy/memory/MEMORY.md`
   - 今日已有的 `📅每日笔记/` 文件（避免重复）
2. **强制读取模板** — `read_file(templates/daily-note每日笔记.md)`，获取完整结构（参见 SKILL.md §2.2，不可跳过）
3. **按模板结构逐节填充** — 禁止凭记忆生成
4. **输出** — `📅每日笔记/YYYY-MM-DD.md`（已存在则追加）

### Prompt 示例
```
用户：day
AI：读取今日已有笔记 + 对话上下文 → 生成/追加每日记录

用户：day 今天学了 React Hooks，遇到 useEffect 依赖问题
AI：以此为核心内容生成今日笔记

用户：day 只记关键事项
AI：生成简化版，只保留关键 bullet points
```

## 2. week / 周复盘

### 触发方式
`week` · `week 本周专注 AI 工具` · `每周` · `周报`

### 执行流程
1. **读取数据源** — 本周所有 `📅每日笔记/`（按日期范围）+ 当前对话 + 全局记忆
2. **强制读取模板** — `read_file(templates/weekly-review每周复盘.md)`，获取完整结构（参见 SKILL.md §2.2，不可跳过）
3. **按模板结构逐节填充** — 汇总关键事项、提取模式/洞察、下周规划
4. **输出** — `📆每周笔记/YYYY-MM-DD-周复盘.md`

### Prompt 示例
```
用户：week
AI：扫描本周所有每日笔记 → 生成周复盘

用户：week 本周主要在做项目复盘
AI：以此为主题生成周复盘，侧重项目相关笔记
```

## 3. note / 主题收录（外部内容摘要）

### 触发方式
`note https://example.com/article` · `note 关于xxx的播客` · `主题` · `收录`

### 执行流程

**步骤1：信息源路由**
- 有 URL → 查询 `references/Extended Toolkit拓展工具包.md` 路由表 → 调用对应 Skill 采集
- 无 URL → 基于用户描述生成

**步骤2：强制读取模板**
- `read_file(templates/note-overview.md)`，获取完整结构（参见 SKILL.md §2.2，不可跳过）
- 按模板结构逐节填充（简要摘要 → 核心内容预览 → 相关资源）— 禁止凭记忆生成

**步骤3：读取 index.md 获取全局视图**
- `read_file(📚主题笔记/index.md)`（不存在则跳过）
- 从 index 快速获取：已有子分类列表、已有笔记及 tags、状态标记
- 用于后续分类匹配和相关资源网络构建，避免全库扫描

**步骤4：自动子分类归档**
- 分析内容主题 → 优先从步骤3的 index 数据匹配已有分类（避免扫描目录）
- 匹配已有分类 → 归入；无匹配 → 新建（**必须 emoji 前缀**）
- 新建时先扫描已有文件夹 emoji 风格保持一致

**步骤5：更新主题索引（index.md）**
- 检查 `📚主题笔记/index.md` 是否存在 → 不存在则创建
- 按笔记 tags 分组追加条目
- 格式：`- [[笔记名]] | 一句话摘要 | YYYY-MM-DD`
- 「最近收录」只保留最近7天

**Index 结构**（`📚主题笔记/index.md`）：
```markdown
# 📚 主题笔记索引

## 📥 最近收录（最近7天）
- [[文章标题]] | 一句话摘要 | YYYY-MM-DD

## 📂 按主题分类（基于 tags）

### AI工具
#### 外部摘要
- [[笔记名]] | 摘要 | YYYY-MM-DD
#### 经验沉淀
- [[笔记名]] | 摘要 | YYYY-MM-DD

## 🔄 状态标记
- 🟡 待整合：[[孤立笔记]]（无入链）
- 🔴 需更新：[[过期笔记]]（超过90天未更新）
```

**步骤6：建立相关资源网络**

优先从步骤3的 index 数据匹配相关笔记（同 tags、同主题），避免全库扫描。

扫描维度：
1. **同主题**：内容主题相似的已有笔记
2. **同作者/来源**：同一作者或平台的其他内容
3. **核心概念**：文中反复提及的关键概念（3次+建议建概念卡片）
4. **相关工具**：文中提到的工具、平台、Skill

输出格式（填入模板 `## 相关资源`）：
```markdown
- 原文: [标题](URL)
- 同主题: [[笔记A]] | [[笔记B]]
- 同作者: [[作者的其他文章]]（如有）
- 核心概念: [[概念_xxx]] → 提及3次+建议建概念卡片
- 相关工具: [[工具名]]（如有）
```

**概念卡片触发条件**：
- 某概念在 ≥3 篇笔记中被提及
- 该概念值得独立成页（有明确定义、应用场景）
- AI 建议创建，用户确认后执行

### Prompt 示例
```
用户：note https://www.example.com/ai-trends
AI：采集文章 → 读取 note-overview 模板 → 判断分类"🤖AI工具" → 归入 📚主题笔记/🤖AI工具/ → 更新 index.md

用户：note 听了 Huberman 关于睡眠的播客
AI：询问要点 → 判断分类为"🏠生活决策" → 归入对应位置
```

## 3b. exp / 经验沉淀（个人实践总结）

### 触发方式
`exp 完成了xxx项目的复盘` · `exp 搭建AI工作流的经验` · `经验` · `沉淀`

### 与 note 的核心区别

| 维度 | note（外部摘要） | exp（经验沉淀） |
|------|---------------|---------------|
| 内容来源 | 外部文章/视频/播客 | 个人实践/复盘/总结 |
| 模板 | note-overview.md | note-experience.md |
| 模板重点 | 文章骨架 + 核心要点 | 实践骨架 + 关键决策 |
| type | note-overview | note-experience |

### 执行流程
同 note 的步骤1-6，但**强制读取** `read_file(templates/note-experience.md)` 模板（参见 SKILL.md §2.2，不可跳过）。

### Prompt 示例
```
用户：exp 完成了Obsidian+AI工作流搭建，总结一下经验
AI：读取 note-experience 模板 → 提取实践步骤 → 归入 📚主题笔记/🤖AI工具/
```

## 4. idea / 创作素材来源

### 触发方式
`idea 这篇文章的观点可以写成小红书` · `idea 对标账号的选题角度不错` · `灵感` · `素材`

### 核心定位
**不是灵感收集或随想**，而是「创作的素材来源」——从别人的内容中标记出可以作为你输出来源的部分。

### 与 note 的区别
| | note | idea |
|--|------|------|
| 目的 | 知识沉淀（学习用） | 输出素材（创作用） |
| 后续动作 | 存档、检索 | 用 `output` 展开为完整创作 |
| 落点 | `📚主题笔记/<🎨子分类>/` | `📚主题笔记/💭创作灵感/` |

### 执行流程
1. 记录素材来源的核心观点 + 标注可输出方向
2. **强制读取模板** — `read_file(templates/note-overview.md)`（**启用「输出方向」章节**），参见 SKILL.md §2.2，不可跳过
3. **读取 index.md** — 获取已有创作灵感条目，避免重复收录
4. 输出到 `📚主题笔记/💭创作灵感/`
5. **更新 index.md** — 在「💭创作灵感」分组下追加条目

### Prompt 示例
```
用户：idea 这篇文章的 xxx 观点可以写成小红书
AI：记录素材来源 + 标注潜在输出方向（小红书）

用户：idea 从xxx视频提取的3个可输出点
AI：批量记录素材来源，每个标注输出方向
```

## 5. output / 创作输出

### 触发方式
`output 写一篇关于xxx的文章` · `output 把xxx主题的笔记整合成总览` · `创作` · `输出`

### 两种场景

**场景 A：原创创作** — 基于用户主题/大纲，可读取相关笔记作参考，输出原创内容
**场景 B：多篇整合** — 扫描 `📚主题笔记/` 相关笔记 → 提取要点 → 合并去重 → 生成结构化总览 → 标注来源 `[[双向链接]]`

### 智能归档
分析内容主题 → 扫描 `📝创作输出/` 已有子分类 → 匹配归入或创建新类别（**emoji 前缀强制**）

### Prompt 示例
```
用户：output 写一篇关于AI笔记工作流的文章
AI：原创创作 → 归入 📝创作输出/🤖AI工具/

用户：output 把AI工具主题的笔记整合成总览
AI：扫描相关笔记 → 合并生成结构化总览 → 📝创作输出/🤖AI工具/
```

## 6. todo / 任务跟踪

### 触发方式
`todo 完成xxx功能` · `todo 调研竞品 p1` · `任务` · `待办` · `task`（兼容旧指令）

### 推荐结构：领域看板

**一个领域一个 md 文件，内部用编号任务 + checkbox 跟踪**——比每个任务单独建文件更轻量。

开箱时会在 `📋待跟踪任务/` 放入样板文件（`🛠️样板-迭代Obsidian Wiki Skill.md`），展示推荐格式，用户可删除。

> 💡 推荐安装 [Tasks 插件](https://publish.obsidian.md/tasks/) 获得打勾自动记录、筛选和看板视图。

### 优先级规则

| 标记 | 含义 | 判断依据 |
|------|------|---------|
| 🔴 P1 | 高优先级 | 紧急且重要，影响核心目标 |
| 🟡 P2 | 中优先级 | 重要但不紧急，或有明确截止日 |
| 🟢 P3 | 低优先级 | 可以延后，不影响主线 |

**AI 判断逻辑**：
1. 用户指令中带 `p1`/`p2`/`p3` → 直接使用
2. 未指定 → AI 根据用户长期记忆、个人配置、近期任务上下文自动判断
3. 无法判断 → 默认 🟡 P2

### 领域看板结构
```markdown
# 🚀 职业发展

### 1. 完成 waytoAGI 作业

🔴 P1 | 2026-04-14 | `#waytoAGI` `#作业`

- [ ] 完成 waytoAGI 作业并提交
- ⏰ 截止 2026-04-16 23:59
- 📎 提交入口：https://waytoagi.com/homework

---

### 2. 更新简历和作品集

🟡 P2 | 2026-04-10 | `#求职` `#作品集`

- [ ] 梳理过去一年的核心项目，整理 3 个可展示的案例
```

**排序规则**：
- **新任务插入到最前面**（倒排序），编号从 1 开始重新排
- 同优先级内按添加时间倒排（最新在最前）

**时间格式**：
- 标题行日期：固定 `YYYY-MM-DD`，为**添加时间**（创建当天），始终展示
- 截止时间（如有）：作为附加信息放在 checkbox 下方，格式 `- ⏰ 截止 YYYY-MM-DD HH:mm`

**附加信息（可选）**：
- checkbox 下方可附加**必要的核心信息**，避免信息噪音
- 适用场景：截止时间、链接、活动时间、关联人物、简要概述
- 格式：`- ⏰ / 📎 / 📅 / 👤 / 💡` + 一行描述
- **原则：只添加"没有就会影响执行"的信息，其他不加**

**结构要点**：
- 数字编号（1、2、3...），**新任务在最前**，编号自动重排
- 每个任务一个 `###` 标题，含：优先级圈圈 + **添加日期** + 标签
- 每个任务一个 checkbox（`- [ ]`），配合 Tasks 插件可打勾
- 截止时间等额外信息放在 checkbox 下方
- 任务之间用 `---` 分隔

### 执行流程
1. 解析任务（名称、截止日期、优先级 P1/P2/P3、附加信息）
2. **强制读取模板** — `read_file(templates/todo任务.md)`，获取领域看板格式（参见 SKILL.md §2.2，不可跳过）
3. **扫描 `📋待跟踪任务/` 已有领域文件**
4. 匹配到已有领域 → **插入到文件最前面**（编号 1），已有任务编号顺延
5. 无匹配领域 → 创建新领域文件（emoji 前缀 + 领域名）
6. 后续可随时更新进度（"完成任务 xxx" → 勾选 checkbox）

### Prompt 示例
```
用户：todo 完成waytoAGI作业 4.16 23:59截止
AI：检测到「🌱个人成长.md」→ 插入为 ### 1（🔴 P1，截止时间紧迫），原任务编号顺延

用户：todo 调研3个竞品，周五前完成
AI：检测到已有「🚀职业发展.md」→ AI 判断为 🟡 P2（有截止日但非紧急），追加

用户：todo 理财规划
AI：无匹配领域 → 创建「💰理财投资.md」新领域文件，默认 🟡 P2
```

## 7. tool / 工具收录·搜索·推荐

### 触发方式
`tool Cursor` · `tool 搜索 用户体验` · `tool 推荐 用户旅程图` · `工具` · `查找`

### 三种模式

**1. 收录模式**：`tool 收录 [工具名]`
- 收录新工具到索引
- 识别类型：Skill → `skills-index.md`；其他 → `🔧tools-index.md`

**2. 搜索模式**：`tool 搜索 [关键词]`
- 检索本地 `🛠️tools/` 文件夹所有文件
- 匹配工具名称、功能描述、标签
- 返回匹配结果列表

**3. 推荐模式**：`tool 推荐 [需求描述]`
- 先在本地索引中搜索相关工具
- 无合适结果时联网推荐合适工具
- 提供工具名称、功能、推荐理由

### 执行流程

**收录模式：**
1. **识别类型**：Skill → `skills-index.md`；其他 → `🔧tools-index.md`
2. **追加表格行** + 生成独立工具卡片
3. **路由联动**：涉及信息采集平台时，更新 `Extended Toolkit拓展工具包.md`

**搜索模式：**
1. 扫描 `🛠️tools/skills-index.md` 和 `🛠️tools/🔧tools-index.md`
2. 匹配工具名称、功能描述、标签中的关键词
3. 返回格式化搜索结果
4. 无结果时提示使用推荐模式

**推荐模式：**
1. 分析需求描述的关键要素
2. 本地搜索相关工具
3. 联网搜索补充推荐（必要时）
4. 按相关性排序返回推荐列表

### 表格模板格式

`skills-index.md`：
```markdown
# Skills Index

## 信息采集
| Skill名称 | 功能 | 推荐 | 路由平台 | 备注 |
|-----------|------|------|---------|------|

## 内容处理
| Skill名称 | 功能 | 推荐 | 备注 |
|-----------|------|------|------|

## 视觉创作
| Skill名称 | 功能 | 推荐 | 备注 |
|-----------|------|------|------|
```

`🔧tools-index.md`：
```markdown
# 🔧 工具索引

## 视频工具
| 工具 | 功能 | 类型 | 推荐 | 备注 |
|------|------|------|------|------|

## 设计工具
| 工具 | 功能 | 类型 | 推荐 | 备注 |
|------|------|------|------|------|
```

### Prompt 示例
```
用户：-tool 收录 Cursor编辑器
AI：识别为工具 → 更新 🔧tools-index.md → 生成工具卡片

用户：-tool 搜索 用户体验
AI：检索本地索引 → 返回相关工具列表

用户：-tool 推荐 用户旅程图工具
AI：本地搜索"旅程图""UX" → 联网补充 → 推荐 UX Journey Gen 等
```

## 9. check / 健康检查

### 触发方式
`check` · `检查` · `健康检查` · `check --skill-only`

### A. 笔记库健康检查

扫描全库，输出 Lint 报告：

```markdown
## 📊 笔记库健康检查报告 | YYYY-MM-DD

### 🔴 孤立笔记（无入链）
| 笔记 | 位置 | 建议关联 |
|------|------|---------|

### 🟠 高频概念待建页
| 概念 | 提及次数 | 建议 |
|------|---------|------|

### 🟡 过期内容（90天未更新）
| 笔记 | 最后更新 | 建议 |
|------|----------|------|

### ⚪ 缺失页面（被引用但不存在）
| 页面名 | 被引用于 | 建议 |
|--------|---------|------|

### 📈 知识网络统计
- 总笔记数 / 孤立笔记 / 概念卡片 / 双向链接覆盖率 / 最近7天新增

### 📋 建议操作（按优先级）
1. [高] ...
2. [中] ...
```

### A2. Index 完整性检查

**快速检查（默认，秒级）**：
1. 读取 `📚主题笔记/index.md`
2. 列出 `📚主题笔记/` 下所有 `.md` 文件名
3. 对比发现：
   - **遗漏条目**：有文件但 index 中未收录 → 自动补入
   - **失效条目**：index 中有但文件已删除 → 自动移除
   - **过期清理**：「最近收录」超过7天的条目 → 移到对应分类区
4. 更新 index.md 的「状态标记」区（孤立笔记、过期笔记）

## 附录B：与 Karpathy LLM Wiki 的对比

### 架构对应

| Karpathy | obsidian-wiki-notes | 说明 |
|----------|-------------------|------|
| Raw Sources | Clippings/ | 原始裁剪，只读不写 |
| Wiki | 📚主题笔记/ | AI 维护的知识层 |
| Schema | SKILL.md | 控制协议 |
| Ingest | `-note` / `-idea` | 收录 / 素材标记 |
| Query | `-output` | 查询沉淀，可触发多篇整合 |
| Lint | `-check` | 健康检查 |
| Index | `📚主题笔记/index.md` | 统一索引 |

### 核心差异

| 维度 | Karpathy Wiki | obsidian-wiki-notes |
|------|---------------|-------------------|
| 目标用户 | 技术用户，喜欢 DIY | 代码小白，开箱即用+自定义 |
| 配置方式 | 工具清单 + prompt | 封装成 Skill，自动化配置 |
| 模板 | 无固定模板，LLM 自由生成 | 结构化模板（可自定义） |
| 索引 | 全局 index.md | 统一 index.md，按 tags 多层分组 |
| 上手门槛 | 需理解工具链 | 5 分钟跑通 |

### 借鉴理念

1. **渐进式编译**：每次 `-note` 不只创建笔记，更更新知识网络（索引、双向链接）
2. **知识复利**：`-output` 多篇整合时，洞察写回主题笔记
3. **用户只读，LLM 维护**：index.md 由 AI 自动维护
4. **健康检查**：模仿 lint，输出结构化报告 + 修复建议

### 为什么保留模板？

Karpathy 无固定模板，LLM 自由生成（灵活但不一致）。obsidian-wiki-notes 选择**结构化模板可自定义**：一致性、可预期、可定制、轻量。

## 10. Emoji 路径安全参考

> 对应 [[obsidian-wiki-notes-skill配置/SKILL]] §2.5。本节提供具体代码示例和排错流程。

### 问题根因

| 问题 | 原因 | 影响 |
|------|------|------|
| 路径找不到文件 | macOS APFS 用 NFD 编码存储 emoji，工具侧期望 NFC | `find`、`grep`、`search_file` 返回空 |
| Shell 命令报错 | emoji 是多字节 UTF-8（4 字节），locale 不匹配时被截断或转义 | `mkdir`、`mv`、`cp` 静默失败 |
| HTML 文件打不开/乱码 | `href`/`src` 中裸写 emoji，浏览器解析异常 | 链接 404、文件名显示 `%EF%BF%BD` |
| 搜索工具返回空结果 | `search_file`/`search_content` 的 glob/regex 引擎不正确处理多字节字符 | 明明存在却搜不到 |

### Shell 安全模板

**创建含 emoji 的目录**：
```bash
export LC_ALL=en_US.UTF-8
VAULT="/path/to/vault"
mkdir -p "$VAULT/📚主题笔记/🤖AI工具"
```

**列出含 emoji 的目录内容**：
```bash
export LC_ALL=en_US.UTF-8
ls -1 "$VAULT/📚主题笔记/"
```

**find 搜索含 emoji 路径下的文件**：
```bash
export LC_ALL=en_US.UTF-8
find "$VAULT/📋待跟踪任务" -name "*.md" -type f
```

**grep 搜索时避免 emoji 在 pattern 中**：
```bash
# ✅ 正确：用 -F 固定字符串匹配
grep -rF "🔴 P1" "$VAULT/📋待跟踪任务/"

# ❌ 错误：emoji 在正则中可能被拆解为多个字节
grep -r "🔴.*P1" "$VAULT/📋待跟踪任务/"
```

### NFC/NFD 规范化

当路径存在但工具找不到时，执行 NFC 规范化：

```bash
# 检查某路径的 Unicode 编码形式
python3 -c "
import unicodedata, sys
path = sys.argv[1]
nfc = unicodedata.normalize('NFC', path)
nfd = unicodedata.normalize('NFD', path)
print(f'原始字节: {path.encode()!r}')
print(f'NFC 字节: {nfc.encode()!r}')
print(f'NFD 字节: {nfd.encode()!r}')
print(f'当前形式: {\"NFC\" if path == nfc else \"NFD\" if path == nfd else \"其他\"}')
" "📚主题笔记"
```

```bash
# 用 NFC 规范化后重试
NFC_PATH=$(python3 -c "import unicodedata,sys; print(unicodedata.normalize('NFC',sys.argv[1]))" "$VAULT/📚主题笔记/")
ls "$NFC_PATH"
```

### AI 工具调用安全路径

AI 在调用内置工具操作含 emoji 的路径时，遵循以下决策树：

```
需要操作含 emoji 的路径？
├─ 读取单个已知文件 → read_file（直接传完整路径，emoji 可用）
├─ 列出目录内容 → 优先 list_dir（兼容性好）
│                  → 失败则 execute_command + ls
├─ 搜索文件
│   ├─ 在 emoji 目录下搜索 → execute_command + find
│   └─ 搜索内容含 emoji → execute_command + grep -rF
├─ 批量操作文件 → execute_command + for 循环，每个路径双引号包裹
└─ 生成 HTML 引用 emoji 路径 → URL 编码 emoji 部分
```

**禁止事项**：
- ❌ `search_file` 的 pattern 参数中包含 emoji
- ❌ `search_content` 的 path 参数使用含 emoji 的中间目录（末级可以）
- ❌ Shell 命令中 emoji 路径不加引号
- ❌ HTML `href`/`src` 中裸写 emoji 字面量

### HTML 中的 emoji 路径编码

```html
<!-- ✅ 正确：URL 编码 -->
<a href="./%F0%9F%93%9A%E4%B8%BB%E9%A2%98%E7%AC%94%E8%AE%B0/index.html">主题笔记</a>

<!-- ✅ 正确：HTML 实体（用于显示文本） -->
<span>&#128218; 主题笔记</span>

<!-- ❌ 错误：裸 emoji 在 href 中 -->
<a href="./📚主题笔记/index.html">主题笔记</a>
```

**Python URL 编码参考**：
```python
from urllib.parse import quote
path = "📚主题笔记/🤖AI工具/file.html"
encoded = quote(path, safe="/")
# → %F0%9F%93%9A%E4%B8%BB%E9%A2%98%E7%AC%94%E8%AE%B0/%F0%9F%A4%96AI%E5%B7%A5%E5%85%B7/file.html
```

### 常见故障排查

| 症状 | 排查步骤 |
|------|---------|
| `search_file` 搜不到 emoji 目录下的文件 | 改用 `execute_command` + `find "路径" -name "*.md"` |
| `mkdir` 创建的文件夹和已有的"看起来一样"但实际是两个 | NFC/NFD 不一致，用 `python3 -c` 检查并统一 |
| HTML 页面点击含 emoji 的链接报 404 | 检查 `href` 是否 URL 编码了 emoji 部分 |
| `grep` 搜索含 emoji 关键词返回空 | 加 `-F` 参数或确认 `LC_ALL=en_US.UTF-8` |
| 自动化脚本在 cron 中找不到 emoji 路径 | cron 不继承用户 locale，在脚本开头加 `export LC_ALL=en_US.UTF-8` |

### C.1 自检清单（笔记创建前必检）

| 检查项 | 验证内容 | 失败处理 |
|--------|---------|---------|
| □ 模板读取 | 是否已 read_file 读取对应模板？结构是否逐节匹配？ | 停止执行，报告"未找到模板" |
